2. Deploy the project (`vercel --prod` or push to the configured branch). Confirm the deployment finishes without build errors.
3. Open the deployed app. Complete the flow:
   - **Connect** with valid Anaplan credentials.
   - **Pick a model** from the picker (prefer a model with ≥ 5 modules but ≤ 30, so the fetch finishes inside 60s).
   - Watch the **s-fetch** screen:
     - Modules counter increments from 0 → total **in batches of 20** — for models with > 20 modules you should see the counter pause briefly between batches as each batch settles before the next begins (BPRT-01)
     - Line items counter rises monotonically (BPRT-02)
     - Current-module text updates as each module finishes (BPRT-02)
4. When the fetch completes, inspect the **schema-preview panel**:
   - `Modules` count is > 0
   - `Line items` count is > 0
   - `Sample module` is a real module name (e.g. `TRK01`, not `—`)
   - `Sample fields` includes at minimum: `id, name, formula, format, appliesTo` (some may be omitted if Anaplan returns them missing — that is acceptable, just note which)
   - `Sample formula` is a non-empty string for a calculated line item (or explicitly `—` if your test model has no formulas; that is also acceptable, just note it)
5. **BPRT-03 (Blob)** — Open the Vercel Blob dashboard. Confirm a new blob exists under `blueprints/{modelId}-{timestamp}.json`. Download it and spot-check that the JSON conforms to `BlueprintDocument`:
   - Top-level: `modelId, workspaceId, fetchedAt, moduleCount, totalLineItems, partialLoad, modules[]`
   - Each module: `id, name, lineItemCount, lineItems[]`
   - Line items contain `formula` text for at least one calculated line item
   - Confirm the raw JSON did NOT come back through the response body (the SSE stream only carries `blobUrl`, never the full document)
6. **BPRT-04 (429/partial-warning)** — Optional but recommended: fetch a larger model (50+ modules) or rapidly re-fetch the same model to provoke a 429. Confirm:
   - Any `partial-warning` events surface as a non-blocking yellow warning strip listing the skipped modules
   - The fetch completes rather than aborting
   - The resulting blob still contains all successful modules; failed modules carry a `fetchError` field
7. **Continue navigation** — Click "Continue to analysis". Confirm the app navigates to the `s-analysis` stub screen showing "Analysis coming in Phase 4." This proves the Phase 3 → Phase 4 handoff is wired correctly.
